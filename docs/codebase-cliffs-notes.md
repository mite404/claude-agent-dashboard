# Codebase Cliffs Notes

Quick-reference tour of the most important functions, types, and data flow.
Use this to understand what fires when, where things live, and how the pieces connect.

---

## The Signal Chain (Big Picture)

Think of the app like a live production monitor. Claude Code is the "talent" on set. Every
time it calls a tool, hook scripts fire, data flows through the Hono API into SQLite, and the
React dashboard polls that database every 2.5 seconds to display it.

```
Claude Code Agent
  │
  ├─ PreToolUse  [Agent] → pre-tool-agent.sh  → POST   /tasks        (creates task)
  ├─ PreToolUse  [*]     → pre-tool-all.sh    → PATCH  /tasks/:id    (pre-event)
  ├─ PostToolUse [*]     → post-tool-all.sh   → PATCH  /tasks/:id    (post-event)
  ├─ PostToolUse [Agent] → post-tool-agent.sh → PATCH  /tasks/:id    (completes task)
  └─ Session events      → session-event.sh   → POST   /sessionEvents

                         Hono :3001  ←  Vite proxy :5173/api/*
                                ↑
                    useTaskPolling(2500ms) polls every 2.5s
```

## Hook Scripts (`scripts/`)

The app's **input layer** — Claude Code calls these on every tool use via `.claude/settings.json`.

| Script | Hook trigger | What it does |
|--------|-------------|--------------|
| `pre-tool-agent.sh` | `PreToolUse [Agent]` | POSTs a new task (`status='running'`) |
| `post-tool-agent.sh` | `PostToolUse [Agent]` | PATCHes task to `completed`/`failed`, appends log |
| `pre-tool-all.sh` | `PreToolUse [*]` | Appends a `HookEvent` (`phase:'pre'`) to the task |
| `post-tool-all.sh` | `PostToolUse [*]` | Updates that HookEvent to `phase:'post'`, sets `completedAt` |
| `session-event.sh` | All 16 session events | POSTs a `SessionEvent` row for every lifecycle event |
| `smoke-test.sh` | Manual | Health check — verifies server + proxy + full CRUD round-trip |
| `spawn-terminal.ts` | HTTP `:3002` | Launches a new `claude` session in a terminal via AppleScript |

### ⚠️ Known gap: HookEvents are silently dropped

`pre-tool-all.sh` and `post-tool-all.sh` try to embed `events: [HookEvent]` in their PATCH
bodies. But `server.ts` strips `events` before every Drizzle UPDATE:

```typescript
const { logs, events, dependencies, ...safeFields } = body;  // events is discarded
```

The `EventTrailRow` in the UI will never show live tool events until this is addressed. Tasks
and session events work correctly — only the per-task tool event trail is affected.

## The API Layer (`src/server.ts`)

Hono REST server on port 3001. All routes are proxied through Vite at `/api/*`.

### Task endpoints

| Method | Path | Called by | Purpose |
|--------|------|-----------|---------|
| `GET` | `/tasks` | `useTaskPolling` | List all; filter by `?status` or `?sessionId` |
| `GET` | `/tasks/:id` | — | Single task by ID |
| `POST` | `/tasks` | `pre-tool-agent.sh` | Create task; auto-upserts `sessions` row |
| `PATCH` | `/tasks/:id` | `patchTask()` | Update task fields (whitelisted) |
| `PUT` | `/tasks/:id` | Legacy shell scripts | Alias for PATCH |
| `DELETE` | `/tasks/:id` | `deleteTask()` | Delete task |

### Session event endpoints

| Method | Path | Called by | Purpose |
|--------|------|-----------|---------|
| `GET` | `/sessionEvents` | `useTaskPolling` | List all; filter by `?sessionId` |
| `POST` | `/sessionEvents` | `session-event.sh` | Create a session event |
| `DELETE` | `/sessionEvents` | `clearAllSessionEvents()` | Delete all (or by sessionId) |

### `handleTaskUpdate` — the PATCH/PUT whitelist

Only these columns are allowed through to the Drizzle UPDATE:

```
name, description, status, kind, priority, progressPercentage,
startedAt, completedAt, claimedAt, agentId, agentType, originatingSkill, taskKind, parentId
```

Everything else (`logs`, `events`, `dependencies`, unknown fields) is stripped. If the
resulting update object is empty, the handler returns the current task as-is (no 500 crash).

## The Database (`src/db/schema.ts`)

SQLite via Drizzle ORM with `casing: 'snake_case'`. TypeScript uses camelCase; the DB uses
snake_case (e.g., `agentType` in code → `agent_type` in SQLite).

### `tasksTable`

The core table. Stored flat — tree structure is built client-side from `parentId`.

| Column | Type | Notes |
|--------|------|-------|
| `id` | text PK | UUID, generated by server |
| `sessionId` | text FK→sessions | Required; auto-upserted by POST /tasks |
| `parentId` | text FK→tasks | Nullable; enables parent-child nesting |
| `name` | text NOT NULL | Task description from hook |
| `status` | text | Default `'unassigned'`; `'blocked'` is never stored |
| `agentId` | text | Hex subagent ID (from SubagentStart event) |
| `agentType` | text | e.g. `'general-purpose'`, `'Explore'` |
| `progressPercentage` | integer | 0–100, default 0 |
| `originatingSkill` | text | e.g. `'/review-pr'` |
| `taskKind` | text | `'orchestrator'` \| `'work'` \| `'background-task'` |

### `sessionEventsTable`

One row per Claude Code lifecycle event. Not task-scoped — covers the whole session.

| Column | Type | Notes |
|--------|------|-------|
| `id` | text PK | UUID |
| `sessionId` | text FK→sessions | |
| `type` | text NOT NULL | One of 16 `SessionEventType` values |
| `summary` | text | Human-readable description |
| `agentId` | text | Which subagent fired this |
| `metadata` | text (JSON) | Parsed to object on read by GET /sessionEvents |

### Other tables

- **`logsTable`** — task execution logs (`id`, `taskId` FK, `timestamp`, `level`, `message`)
- **`sessionsTable`** — session records (`id`, `type`, `status`, `agentType`, `model`)
- **`taskDependenciesTable`** — dependency edges (`taskId`, `dependsOnId`, composite PK)

**Why `logs` isn't in the task row:** Logs are stored in `logsTable` (one row per entry),
not embedded in the task. `GET /tasks` does NOT return a `logs` field. The `Task` type
has `logs?: LogEntry[]` as optional for exactly this reason.

## Frontend Data Fetching (`src/hooks/useTaskPolling.ts`)

The heartbeat of the dashboard. Everything the UI shows flows through here.

### `useTaskPolling(intervalMs = 2500): UseTaskPollingResult`

```
Returns: { tasks, tree, sessionEvents, loading, lastUpdated, error, refresh }
```

**On each poll cycle:**

1. Fires `GET /api/tasks` and `GET /api/sessionEvents` in parallel
2. Calls `computeBlockedState(tasks)` — mutates status to `'blocked'` in-place
3. Calls `buildTree(tasks)` — converts flat array to nested `TaskNode[]`
4. Updates React state → triggers re-render

Called by: `Dashboard.tsx`

### `computeBlockedState(tasks: Task[]): void`

Mutates the task array before tree-building. For each task with `dependencies`, if any
dependency has a status other than `'completed'` or `'cancelled'`, the task's `status` is
set to `'blocked'` and `blockedBy` is populated with the blocking IDs.

`'blocked'` is **never stored in SQLite** — computed fresh on every poll.

### `buildTree(tasks: Task[]): TaskNode[]`

1. Creates a `Map<id, TaskNode>` — each task gets `children: []`
2. Second pass: if task has a `parentId` that exists in the map, push into parent's `children`
3. Orphaned children (missing parent) are promoted to roots
4. Returns root nodes only — children are nested inside

```

                 └→ render <TaskRow> for each FlatTask
```

## Session Events Strip (`src/components/GlobalEventStrip.tsx`)

Collapsible panel at the bottom of the dashboard. Props: `events: SessionEvent[]` (from
`useTaskPolling` → `TaskTable` → `GlobalEventStrip`) and `onClearAllEvents?: () => Promise<void>`.

- Auto-scrolls to the newest event when the panel is open
- Shows emoji, event type name, summary (truncated with title tooltip), agent ID, timestamp
- Shows a violet skill pill for `UserPromptSubmit` events that have `originatingSkill` set
- "Clear all" button calls `onClearAllEvents` prop → `clearAllSessionEvents()` in taskApi.ts
- Starts **open** if `events.length > 0`, **collapsed** if empty

## TypeScript Types (`src/types/task.ts`)

### `Task` — the core model

```typescript
interface Task {
  id: string;
  name: string;
  status: TaskStatus;        // see union below; 'blocked' is computed, never stored
  agentType: string;         // 'general-purpose' | 'Explore' | etc.
  parentId?: string | null;  // enables tree nesting
  sessionId?: string;
  createdAt: string;         // ISO timestamp
  startedAt?: string | null;
  completedAt?: string | null;
  progressPercentage: number;
  logs?: LogEntry[];         // optional — stored in logsTable, not in GET /tasks response
  events?: HookEvent[];      // tool events — currently not persisted (see known gap)
  dependencies?: string[];   // task IDs this task waits for
  agentId?: string;          // hex subagent ID from SubagentStart
  originatingSkill?: string; // e.g. '/review-pr'
  taskKind?: TaskKind;
}

type TaskStatus = 'running' | 'completed' | 'failed' | 'paused'
                | 'cancelled' | 'blocked' | 'unassigned';
```

### `TaskNode` — frontend-only tree form

```typescript
interface TaskNode extends Task {
  children: TaskNode[];  // nested child tasks, built by buildTree()
  blockedBy?: string[];  // computed by computeBlockedState(), never in DB
}
```

### `SessionEvent` — lifecycle event (not task-scoped)

Key fields: `id`, `type: SessionEventType`, `timestamp`, `sessionId`, `summary`,
`agentId?`, `agentType?`, `originatingSkill?`, `metadata?: Record<string, unknown>`

### `HookEvent` — a single tool call event

```typescript
interface HookEvent {
  id: string;          // same as tool_use_id from Claude Code
  toolName: string;    // 'Bash' | 'Read' | 'Write' | 'Edit' | etc.
  phase: 'pre' | 'post';
  status: 'running' | 'completed' | 'failed';
  summary: string;     // first ~120 chars of tool input
  timestamp: string;
  completedAt?: string;
}
```

## Utility Functions

### `src/lib/utils.ts`

- **`cn(...inputs)`** — combines `clsx` + `tailwind-merge`. Used in every component for
  conditional class name composition.
- **`formatElapsed(startedAt?, completedAt?)`** — e.g. `"2m 34s"` or `"—"`. Used in
  TaskRow duration column and CheckpointRow.
- **`formatTimestamp(iso)`** — converts ISO string to `HH:MM:SS` (24hr). Used in log
  rows and GlobalEventStrip.

### `src/lib/taskUtils.ts`

- **`sortNodes(nodes, sort)`** — recursively sorts the task tree by column. Preserves
  parent-child relationships at every depth level. Used in TaskTable's `useMemo` chain.
- **`flattenVisible(nodes, expanded, depth)`** — converts the tree into a flat array for
  rendering, skipping collapsed subtrees. Returns `FlatTask[]` with `depth` for indentation.
- **`collectAllTasks(nodes)`** — recursively collects every task node into a flat array.
  Used for bulk actions and the "has completed tasks?" check.
- **`collectIds(nodes)`** — recursively collects all task IDs. Used to track newly-arrived
  tasks for the highlight animation.
